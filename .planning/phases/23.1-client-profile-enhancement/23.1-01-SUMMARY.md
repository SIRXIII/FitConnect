# Phase 23.1 Plan 01: DB Migration + Shared Constants + Zod Schema Extension Summary

**One-liner:** SQL migration adding 3 client_profiles columns with booking_requests RLS, plus profileConstants.ts and standalone Zod field schemas for Phase 23.1 UI plans.

## What Was Built

### Task 1: Database Migration

Migration `20260319200000_client_profile_enhancement.sql` adds:

- `health_conditions jsonb NOT NULL DEFAULT '[]'` — structured checklist of health conditions
- `intensity_preference text CHECK ('light', 'moderate', 'intense')` — nullable workout intensity preference
- `goals_ranked jsonb NOT NULL DEFAULT '[]'` — ordered top-3 goals array
- COMMENT ON COLUMN for all 3 new columns
- RLS policy: "Trainers can view client profiles for their booking requests" — allows trainer read access via pending `booking_requests` rows (pre-booking queue, complementing the existing bookings-based policy)

### Task 2: Shared Constants + Zod Schemas

**`src/lib/profileConstants.ts`** (new file, 8 named exports):
- `FITNESS_GOALS` — 7 goal objects with value/label
- `WORKOUT_TYPES` — 12 workout type objects
- `FREQUENCIES` — 4 frequency options
- `HEALTH_CONDITIONS` — 6 health condition objects (new)
- `INTENSITY_LEVELS` — 3 intensity level objects with color classes (new)
- `HEALTH_CONDITION_VALUES` — value-only array for Zod
- `INTENSITY_LEVEL_VALUES` — `['light', 'moderate', 'intense'] as const`
- `FITNESS_GOAL_VALUES` — value-only array for Zod

**`src/lib/schemas.ts`** extensions (Phase 23.1 section):
- `healthConditionsSchema` — `z.array(z.string()).default([])`
- `intensityPreferenceSchema` — `z.enum(INTENSITY_LEVEL_VALUES).optional()`
- `goalsRankedSchema` — `z.array(z.string()).max(3).default([])`
- Type exports: `HealthConditionsInput`, `IntensityPreferenceInput`, `GoalsRankedInput`

## Decisions Made

1. **Standalone field schemas** — new Zod schemas are independent, not merged into `fitnessPassportSchema`. UI plans will use per-field auto-save patterns, not a monolithic form submit.

2. **Value array cast pattern** — `HEALTH_CONDITION_VALUES = HEALTH_CONDITIONS.map(c => c.value) as unknown as readonly [string, ...string[]]` avoids brittle Zod tuple type inference from `.map()` while remaining enum-compatible.

3. **Pre-existing TS errors** — RadiusCircle, AnalyticsTab, PayoutsTab, and ReferralLeaderboard have pre-existing TypeScript errors unrelated to this plan. No new errors introduced by these changes.
