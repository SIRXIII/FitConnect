# Phase 23.1: Client Profile Enhancement - Research

**Researched:** 2026-03-19
**Domain:** React form patterns, Supabase schema extension, drag-to-rank UX, auto-save, progress completion ring
**Confidence:** HIGH

## Summary

This phase extends the existing `ClientPassport.tsx` page into a full-featured profile hub. The technical domain is entirely within the existing FitRush stack — no new libraries are needed. The implementation work is primarily: (1) a DB migration adding three new columns to `client_profiles`, (2) refactoring `ClientPassport.tsx` into scrollable labeled sections with auto-save-on-blur per field, (3) four new UI controls (progress ring, health checklist, intensity slider, drag-to-rank goals), and (4) augmenting `BookingRequestCard.tsx` with client summary data from the new fields.

The existing RLS on `client_profiles` already covers trainer read access via a `bookings` join. The `booking_requests` flow also needs trainer access to client profile data — this requires a new policy or RLS extension. The current RLS policy is `bookings`-based only; since the Go Live request queue fires before a booking is created, trainers accepting requests cannot yet read client_profiles under existing policy.

**Primary recommendation:** Add new DB columns via migration, refactor ClientPassport into a single-page auto-save experience, implement controls as standalone sub-components, and extend RLS to cover `booking_requests` (not just `bookings`).

### Pattern 1: Auto-save on Blur (Field-level Upsert)
**What:** Each form section (or field group) saves independently on `onBlur`. No master save button. Sonner toast shows "Saved" briefly.
**When to use:** Every field in Personal Info, Health, and Preferences sections.
**Example:**
```typescript
// Pattern used throughout ClientPassport already for avatar
const handleBlurSave = async (field: string, value: unknown) => {
  if (!user) return;
  const { error } = await supabase
    .from('client_profiles')
    .upsert({ user_id: user.id, [field]: value }, { onConflict: 'user_id' });
  if (!error) toast.success('Saved', { duration: 1500 });
};
```
**Debounce timing:** 300ms is sufficient for text areas (avoids a save on every keypress during typing). For pill/button toggles, save immediately on click since there is no intermediate state.

### Pattern 2: Progress Ring (SVG Circle)
**What:** An SVG `<circle>` with `strokeDasharray` / `strokeDashoffset` to visually encode completion. No external library needed.
**When to use:** At top of ClientPassport page.
**Example:**
```typescript
// r = 36, circumference = 2 * Math.PI * 36 ≈ 226
const CIRCUMFERENCE = 226;
const offset = CIRCUMFERENCE - (completionPct / 100) * CIRCUMFERENCE;
<circle
  cx="44" cy="44" r="36"
  fill="none"
  stroke="currentColor"
  strokeWidth="4"
  strokeDasharray={CIRCUMFERENCE}
  strokeDashoffset={offset}
  strokeLinecap="round"
  className="transition-all duration-500 text-accent"
/>
```
**Completion fields to track:** avatar, age, weight, height, fitness_level, health_conditions (any selection), intensity_preference, goals_ranked (at least 1).

### Pattern 3: 3-Stop Intensity Slider
**What:** Three clickable stops (not a native `<input type="range">` — native range gives poor mobile UX and is hard to style). Three button-pills in a row with a connecting track, color-coded green/amber/red.
**When to use:** Preferences section for intensity_preference.
**Example:**
```typescript
const STOPS = [
  { value: 'light', label: 'Light', desc: 'Focus on form and mobility', color: 'text-green-600 border-green-400' },
  { value: 'moderate', label: 'Moderate', desc: 'Balanced effort and challenge', color: 'text-amber-500 border-amber-400' },
  { value: 'intense', label: 'Intense', desc: 'Push limits and max output', color: 'text-red-500 border-red-400' },
];
```
A horizontal track with the three stops marked; clicking a stop fills the track from left to that stop.

### Pattern 4: Drag-to-Rank Goals (Button-based Reorder)
**What:** Show selected goals in a ranked list with up/down arrow buttons. This is simpler and more reliable on mobile than native HTML5 drag-and-drop (which has poor touch support) or a library like `@dnd-kit/core`. The user sees their top 3 ranked goals with move-up/move-down chevron buttons. Remaining unselected goals appear below as an add pool.
**Rationale for button-based over native drag:** Mobile-first app, native drag events are unreliable on iOS Safari. Button reorder is accessible, zero-dependency, and visually clear.
**When to use:** Preferences section for goals_ranked.

### Pattern 5: ClientSummaryCard (Trainer View)
**What:** A condensed read-only card shown inside `BookingRequestCard` and on booking detail. Shows: avatar, name, fitness level badge, top goal, health condition icons, intensity color indicator.
**When to use:** Inside `BookingRequestCard` (existing component), and in `BookSession` detail view.
**Expand behavior:** The card should be tappable to expand to show all fields. Framer Motion `AnimatePresence` handles expand/collapse.

### Pattern 6: Scrollable Sections (No Tabs)
**What:** One long scroll page with `<section>` blocks labeled "Personal Info", "Health", "Preferences". A sticky small section label or visible heading identifies each block as user scrolls.
**When to use:** The entire ClientPassport page layout.

### Anti-Patterns to Avoid
- **Full-page save button:** The new design is auto-save-on-blur per section. Keep the existing avatar upload pattern (immediate save). Remove the master "Save Fitness Passport" button, or scope it only to fields that don't naturally trigger blur.
- **Native `<input type="range">` for intensity:** Styling cross-browser is brittle. Use three button stops instead.
- **HTML5 drag-and-drop for mobile:** `dragstart`/`drop` events are unreliable on iOS Safari. Use arrow button reorder.
- **Fetching client_profiles inside BookingRequestCard directly:** The parent component (Go Live request queue) should fetch and pass client profile data as a prop to keep BookingRequestCard stateless.
- **Saving goals_ranked as text[]:** Store as `jsonb` so order is preserved. `text[]` in Postgres does preserve order, but `jsonb` array is more explicit for ranked ordered data.

## Database Changes Required

### New Migration
Three new columns on `client_profiles`:

```sql
-- health_conditions: structured checklist selections (jsonb array of string values)
ALTER TABLE public.client_profiles
  ADD COLUMN IF NOT EXISTS health_conditions jsonb NOT NULL DEFAULT '[]'::jsonb;

-- intensity_preference: one of light, moderate, intense
ALTER TABLE public.client_profiles
  ADD COLUMN IF NOT EXISTS intensity_preference text
    CHECK (intensity_preference IN ('light', 'moderate', 'intense'));

-- goals_ranked: ordered array of up to 3 goal values (jsonb to preserve order)
ALTER TABLE public.client_profiles
  ADD COLUMN IF NOT EXISTS goals_ranked jsonb NOT NULL DEFAULT '[]'::jsonb;
```

### RLS Extension (CRITICAL)
The existing trainer read policy is:
```sql
CREATE POLICY "Trainers can view client profiles for their bookings"
  ON public.client_profiles FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.bookings b
      JOIN public.trainer_profiles tp ON tp.id = b.trainer_id
      WHERE b.client_id = client_profiles.user_id
        AND tp.user_id = auth.uid()
    )
  );
```
This only covers trainers with an existing `bookings` row. The Go Live request queue fires from `booking_requests` (before a booking exists). Trainers need to read client profiles from the request queue to prepare session plans.

**Fix needed — new policy:**
```sql
CREATE POLICY "Trainers can view client profiles for their booking requests"
  ON public.client_profiles FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.booking_requests br
      JOIN public.trainer_profiles tp ON tp.id = br.trainer_id
      WHERE br.client_id = client_profiles.user_id
        AND tp.user_id = auth.uid()
        AND br.status = 'pending'
    )
  );
```

## Common Pitfalls

### Pitfall 1: Auto-save Floods Supabase on Text Input
**What goes wrong:** Saving `health_notes` or `bio` on every `onChange` event fires a DB write for every keystroke.
**Why it happens:** Developers conflate "auto-save" with "save on every change".
**How to avoid:** Use `onBlur` (not `onChange`) for textarea fields. For number inputs (age, weight, height), save on `onBlur`. For button/pill selectors, save immediately on click since selection is discrete.
**Warning signs:** 429 rate-limit errors from Supabase; unusually high DB write count in dashboard.

### Pitfall 2: RLS Blocks Trainer from Reading Client Profile in Request Queue
**What goes wrong:** Trainer clicks a booking request, the client summary card shows blank data even though the client has a complete profile.
**Why it happens:** Existing RLS policy only grants trainer read access via `bookings` table. At request time, no `bookings` row yet exists.
**How to avoid:** Add the new policy covering `booking_requests` as described in Database Changes above.
**Warning signs:** Supabase returns `{}` or empty array for client_profiles query when trainer is authenticated.

### Pitfall 3: goals_ranked vs fitness_goals Out of Sync
**What goes wrong:** The old `fitness_goals text[]` column (multi-select, up to 5) and the new `goals_ranked jsonb` (ordered top 3) conflict. A client could have goals selected in `fitness_goals` that aren't in `goals_ranked` and vice versa.
**Why it happens:** The phase adds a new ranked column without defining the relationship to the old column.
**How to avoid:** Treat `goals_ranked` as the authoritative field going forward. The old `fitness_goals` column continues to exist (backward compatibility with onboarding) but the ClientPassport edit UI surfaces only `goals_ranked`. The AI matching phase (25) should use `goals_ranked` for primary goal.

### Pitfall 4: health_conditions jsonb Schema Inconsistency
**What goes wrong:** Different parts of the app save different shapes to `health_conditions` — some as array of strings, some as array of objects.
**Why it happens:** No single enforced schema for jsonb.
**How to avoid:** Define a canonical format in code and document it. Use string array: `["back_pain", "asthma"]`. The Zod schema extension validates this.

### Pitfall 5: Progress Ring Counts Fields Missing Before DB Load
**What goes wrong:** The ring momentarily shows 0% on page mount before data loads from Supabase, causing a flash.
**Why it happens:** Completion is computed from local state, which is empty before `useEffect` fills it.
**How to avoid:** Start completion calculation after the initial `load` state clears. Keep the existing skeleton loading pattern (already in ClientPassport) to delay rendering the ring until data is ready.

### Pitfall 6: Framer Motion Drag on Mobile (if native drag attempted)
**What goes wrong:** `useDrag` / HTML5 drag events do not fire reliably on iOS Safari for touch events.
**Why it happens:** iOS Safari does not implement `dragstart` on touch; requires `touchstart`/`touchmove` hacks.
**How to avoid:** Use button-based reorder (up/down arrows) — fully accessible and cross-platform reliable.

## Code Examples

### Zod Schema Extension
```typescript
// In src/lib/schemas.ts — extend fitnessPassportSchema
const HEALTH_CONDITIONS = [
  'back_pain', 'knee_injury', 'asthma', 'heart_condition', 'pregnancy', 'diabetes',
] as const;

const INTENSITY_LEVELS = ['light', 'moderate', 'intense'] as const;

export const fitnessPassportSchemaV2 = fitnessPassportSchema.extend({
  health_conditions: z.array(z.enum(HEALTH_CONDITIONS)).default([]),
  health_notes_other: z.string().max(1000).optional(),
  intensity_preference: z.enum(INTENSITY_LEVELS).optional(),
  goals_ranked: z.array(z.string()).max(3).default([]),
});
```

### Auto-save Hook (reusable)
```typescript
// Pattern for ClientPassport auto-save on blur
const saveField = async (updates: Partial<ClientProfileUpdate>) => {
  if (!user) return;
  const { error } = await supabase
    .from('client_profiles')
    .upsert({ user_id: user.id, ...updates }, { onConflict: 'user_id' });
  if (!error) toast.success('Saved', { duration: 1200, id: 'profile-save' });
};
// Using { id: 'profile-save' } prevents toast stacking on rapid blur events
```

### Progress Ring Completion Computation
```typescript
const COMPLETION_FIELDS = [
  !!avatarPreview,              // avatar
  !!age,                        // age
  !!weightLbs,                  // weight
  !!(heightFt || heightIn),     // height
  !!fitnessLevel,               // fitness_level
  healthConditions.length > 0 || healthNotesOther.trim().length > 0, // health
  !!intensityPreference,        // intensity
  goalsRanked.length > 0,       // goals
] as const;
const completionPct = Math.round(
  (COMPLETION_FIELDS.filter(Boolean).length / COMPLETION_FIELDS.length) * 100
);
```

### ClientSummaryCard Shape (what BookingRequestCard needs)
```typescript
interface ClientSummaryData {
  full_name: string;
  avatar_url: string | null;
  fitness_level: 'beginner' | 'intermediate' | 'advanced' | null;
  primary_goal: string | null;       // goalsRanked[0]
  health_conditions: string[];       // icons/badges
  intensity_preference: 'light' | 'moderate' | 'intense' | null;
}
```

### Fetching Client Profile for Trainer (booking_requests join)
```typescript
// In Go Live request queue, when loading pending requests:
const { data } = await supabase
  .from('booking_requests')
  .select(`
    id, client_id, slot_id, status, created_at,
    client:profiles!client_id(full_name, avatar_url),
    client_profile:client_profiles!client_id(
      fitness_level, goals_ranked, health_conditions, intensity_preference
    ),
    slot:availability_slots!slot_id(start_time, end_time)
  `)
  .eq('trainer_id', trainerProfileId)
  .eq('status', 'pending');
```
Note: Supabase JS v2 supports nested select with `!foreign_key` syntax for disambiguation when multiple FK relationships exist.

## Open Questions

1. **Should `fitness_goals` be deprecated or kept alongside `goals_ranked`?**
   - What we know: Both columns will exist after this phase. Onboarding writes to `fitness_goals`. ClientPassport will write to `goals_ranked`.
   - What's unclear: AI matching (Phase 25) will use this data. Which column should it use?
   - Recommendation: Keep both. Document that `goals_ranked` is the priority source. Phase 25 research can decide whether to migrate or unify.

2. **Expand/collapse behavior of ClientSummaryCard in BookingRequestCard**
   - What we know: User decision says "details available on tap/expand".
   - What's unclear: Whether the expanded view should be an inline expand (same card grows) or a modal/sheet.
   - Recommendation: Inline expand via Framer Motion `AnimatePresence` for card height. Keeps mobile context without navigation disruption.

3. **Is the existing `"Trainers can view client profiles for their bookings"` policy sufficient for booking detail page?**
   - What we know: The booking detail page uses an existing `bookings` row, so the existing policy covers it.
   - What's unclear: Nothing — the new policy is only needed for the pre-booking `booking_requests` queue.
   - Recommendation: Add only the one new policy covering `booking_requests`.
