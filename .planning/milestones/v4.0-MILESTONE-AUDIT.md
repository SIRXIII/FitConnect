### Missing Connections

#### GAP-01: CALSYNC-02 — sync-booking-to-gcal has no trigger/caller

**Severity:** HIGH
**Requirement:** CALSYNC-02 — FitRush bookings automatically push to Google Calendar as events

The `sync-booking-to-gcal` Edge Function was built in Phase 28 Plan 03 and exists at:
`supabase/functions/sync-booking-to-gcal/index.ts`

However, **nothing calls it**. A search for `sync-booking-to-gcal` across the entire codebase (src/, supabase/functions/, migrations/) returns zero results outside the function directory itself and planning docs.

Expected callers (none present):
- `BookSession.tsx` — should invoke after `create_booking_atomic` succeeds
- `BookingRequestQueue.tsx` — should invoke after trainer accepts a request
- `stripe-webhook/index.ts` — could fire on `payment_intent.succeeded`
- A DB trigger with `pg_net` on bookings INSERT (no such trigger exists)
- pg_cron for batch sweep (only sync-gcal-events is scheduled, not sync-booking-to-gcal)

**Evidence of break:** Phase 28-03 SUMMARY.md confirms the function "accepts both service-role auth (for automated triggers post-booking) and user auth (manual trigger)" — but the automated trigger was never added. The SUMMARY notes the function "Silently skips trainers with no active GCal connection" and is "idempotent" — these are design features for a function that was intended to be called, not a function that calls itself.

**Impact:** Booking events are never pushed to Google Calendar. CALSYNC-02 is wired in the Edge Function but unreachable. The iCal fallback (CALSYNC-05) continues to work unaffected.

### Broken Flows

#### FLOW-BREAK-01: Trainer connects Google Calendar → booking syncs

**Steps complete:**
1. Trainer navigates to TrainerDashboard Calendar tab — WIRED (GoogleCalendarConnect rendered above CalendarExportCard)
2. Trainer clicks Connect — WIRED (OAuth popup opens via useGcalConnection.connect())
3. OAuth popup completes → GoogleCalendarCallback posts code to opener — WIRED
4. google-calendar-connect Edge Function exchanges code, stores tokens — WIRED
5. Client books a session → `create_booking_atomic` RPC succeeds — WIRED
6. **Booking event pushed to Google Calendar — BROKEN** (sync-booking-to-gcal never called)
7. Booking cancellation removes GCal event — WIRED (cancel-booking Edge Function extended in 28-03)

The break is at step 6. Steps 7 and the inbound direction (external events blocking slots) work correctly because `sync-gcal-events` is scheduled via pg_cron.

#### FLOW-04: Trainer connects Google Calendar → cancellation removes event

Steps 1-5 as FLOW-BREAK-01 above.
6. Booking event push — BROKEN (see FLOW-BREAK-01)
7. Client cancels booking: `cancel-booking` Edge Function fires → fetches `gcal_event_id` from booking — WIRED (Phase 28-03)
8. `deleteGcalEvent` called via `gcal-helpers.ts`, best-effort (non-blocking) — WIRED (Phase 28-03)

**Note:** Step 7-8 would only delete the event if step 6 had run first and stored a `gcal_event_id`. With step 6 broken, `gcal_event_id` on the booking is always null, so the GCal deletion step in cancel-booking silently no-ops. The cancellation itself succeeds regardless.

4. **Google OAuth consent screen verification** — External 4-8 week Google approval process started in Phase 21 (GCP-SETUP-CHECKLIST.md). GCal sync features will not reach production users until approved. Feature exists in code; production deployment is gated on Google's approval, not on code.

5. **pg_cron/pg_net availability on free tier** — Both availability expiry (Phase 22) and GCal polling (Phase 28) use graceful RAISE NOTICE fallbacks if extensions unavailable. Functional on paid Supabase plans.

## Final Verdict

**GAPS_FOUND**

The milestone is 43/44 requirements fully wired. One gap requires remediation before CALSYNC-02 can be considered complete:

**Required fix:** Wire `sync-booking-to-gcal` Edge Function as a caller from the booking confirmation path. The most appropriate integration point is in `BookSession.tsx` immediately after `create_booking_atomic` returns a `booking_id`, and in `BookingRequestQueue.tsx` after the trainer accepts a request (which also calls `create_booking_atomic`). The Edge Function itself is correctly built, handles idempotency, and gracefully skips trainers without GCal connections.

**All other 43 requirements are fully integrated and verified.**
