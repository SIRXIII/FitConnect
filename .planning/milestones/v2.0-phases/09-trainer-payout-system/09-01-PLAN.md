**create-payout/index.ts** -- New Edge Function following the project pattern (Deno.serve, corsHeaders, requireEnv, bearer token auth):

1. **Auth:** Verify trainer JWT using the userClient pattern from create-connect-account. Get user.id.
2. **Fetch trainer profile:** Query trainer_profiles where user_id = user.id. Extract stripe_account_id. If no stripe_account_id, return 400 "Stripe account not connected."
3. **Calculate available balance:** Query payments table:
   ```sql
   SUM(payments.trainer_payout) WHERE payments.status = 'succeeded'
     AND payments.payout_transaction_id IS NULL
     AND payments.booking_id IN (SELECT id FROM bookings WHERE trainer_id = trainer_profile.id)
   ```
   Use the Supabase client. The trainer_payout column already has fees deducted (PAYOUT-04 -- do NOT recompute fees).
4. **Guard: balance < $50** -> return 400 with `{ error: "Minimum payout amount is $50" }`.
5. **Guard: existing pending payout** -> Query payout_transactions WHERE trainer_id AND status IN ('pending', 'processing'). If exists, return 409 with `{ error: "A payout is already in progress" }`.
6. **Begin payout flow** (use adminClient with service role for DB writes):
   a. INSERT into payout_transactions: trainer_id, amount (the available balance), status: 'pending', initiated_by: 'trainer'. Get the inserted row ID.
   b. UPDATE payments SET payout_transaction_id = {new row ID} WHERE the same filter used in step 3 (the exact same payments being swept).
   c. Call `stripe.transfers.create({ amount: Math.round(balance * 100), currency: 'usd', destination: stripe_account_id, metadata: { trainer_id, payout_transaction_id } })`.
   d. UPDATE payout_transactions SET status = 'processing', stripe_transfer_id = transfer.id.
   e. If Stripe call fails: UPDATE payout_transactions SET status = 'failed', UPDATE payments SET payout_transaction_id = NULL (rollback sweep). Return 502 with error message. Handle `insufficient_funds` Stripe error explicitly with a 402.
7. **Send initiation email:** Call `send-notification-email` Edge Function internally OR call Resend API directly. Use Resend directly (simpler -- per research anti-pattern advice). Fetch trainer's email from profiles table. Send with:
   - from: 'FitRush <noreply@resend.dev>' (use resend.dev for dev, per research)
   - subject: 'Your FitRush payout has been initiated'
   - html: `<p>Your payout of $${amount.toFixed(2)} has been initiated. Funds expected within 2 business days.</p>`
   - Use RESEND_API_KEY env var.
   Email failure should NOT fail the payout -- log warning and continue.
8. **Return** 200 with `{ success: true, amount, transferId: transfer.id }`.

Use `npm:stripe@14.25.0` and `https://esm.sh/@supabase/supabase-js@2.49.8` imports. Use apiVersion '2023-10-16'. Use Stripe.createFetchHttpClient().

**send-notification-email/index.ts** -- Replace the console.log stub with Resend API:

Keep the existing interface (to, subject, body fields). Keep the existing auth check. Replace the `console.log` block with:
```typescript
const resendApiKey = Deno.env.get('RESEND_API_KEY');
if (!resendApiKey) {
  // Fallback to logging if no API key configured (dev mode)
  console.log('[send-notification-email] No RESEND_API_KEY, logging:', { to, subject });
} else {
  const res = await fetch('https://api.resend.com/emails', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${resendApiKey}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      from: 'FitRush <noreply@resend.dev>',
      to: [body.to],
      subject: body.subject,
      html: body.body,
    }),
  });
  if (!res.ok) {
    const errBody = await res.text();
    console.error('[send-notification-email] Resend error:', res.status, errBody);
    // Still return success to caller -- email failure should not block operations
  }
}
```
Keep the same response format. This makes the function backwards-compatible with all existing callers while adding real email delivery.
