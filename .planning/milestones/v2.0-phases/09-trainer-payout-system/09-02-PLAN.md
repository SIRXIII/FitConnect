**PayoutsTab.tsx -- New component with these sections:**

**A. Balance Display (PAYOUT-01, PAYOUT-04):**
- Hero available balance: Large number using the stat card pattern but bigger (text-5xl serif). Label: "Available Balance". Format as USD ($X,XXX.XX).
- Pending balance below: Smaller (text-lg), label "Pending" with a tooltip (title attribute): "Completed sessions not yet paid out".
- No fee breakdown shown -- just the net withdrawable amount per user decision.
- **Balance query -- client-side aggregation approach:** Supabase JS v2 does not support SUM aggregations natively. Fetch the rows and reduce client-side:
  - Available: Use supabase client to query `payments` with `.select('trainer_payout')` filtered by `.eq('status', 'succeeded')` and `.is('payout_transaction_id', null)`. Join with bookings via `.in('booking_id', trainerBookingIds)` where trainerBookingIds is fetched first from `bookings` table `.select('id').eq('trainer_id', trainerProfile.id)`. Then reduce the result array: `data.reduce((sum, row) => sum + Number(row.trainer_payout), 0)`.
  - Pending: Same approach but filter payments by `.in('status', ['pending', 'processing'])` instead of 'succeeded'. Same client-side reduce.
- Use a Supabase realtime subscription on `payments` table filtered by trainer's bookings to auto-update balance when new payments come in.
- Show a loading skeleton (pulsing gray bars) while loading.

**B. Request Payout Button (PAYOUT-02):**
- "Request Payout" button below balance cards.
- **Disabled state** when available balance < $50: button has `disabled` attribute + `title="Minimum $50 required"` tooltip. Style: opacity-50 cursor-not-allowed.
- **Enabled state:** Standard accent border button matching project style.
- On click: Open a confirmation modal (simple div overlay with backdrop, matching project aesthetic -- border border-ink/10, bg-paper).
- Modal copy: "Request payout of $X.XX? Funds arrive within 2 business days." with Confirm and Cancel buttons.
- **On Confirm:**
  - Set loading state on button
  - `fetch(${SUPABASE_URL}/functions/v1/create-payout, { method: 'POST', headers: { Authorization: Bearer ${session.access_token} } })`
  - On success (200): `toast.success('Payout requested successfully')`, close modal, set available balance to $0 immediately (optimistic), refetch balance.
  - On error: `toast.error('Payout failed. Please try again or contact support.')` per user decision. Close modal. Balance stays as-is.

**C. Transaction History Table (PAYOUT-05):**
- Query both:
  1. `payments` joined with bookings (for booking credit rows) -- columns: created_at as date, client name from profiles, trainer_payout as amount, status.
  2. `payout_transactions` (for transfer rows) -- columns: created_at as date, "Payout Transfer" as client, amount, status.
- Combine into a single array, sort by date DESC (newest first per user decision).
- Table columns: **Date | Client | Amount | Status** per user decision.
- Date format: short date (e.g., "Mar 14, 2026").
- Amount: formatted as USD.
- Status badges using existing pattern:
  - **Completed/succeeded**: green dot + "Completed" text
  - **Pending/processing**: amber dot + "Pending" text
  - **Failed**: red dot + "Failed" text
- No filtering -- per user decision, filtering deferred to Phase 10.
- Net payout per row only -- no per-row fee breakdown per user decision.
- If no transactions: Empty state message "No transactions yet."
